## 1. Next.js 16 Performance Best Practices

### App Router Optimization
- **Partial Prerendering (PPR)** is stable in Next.js 16. It allows static shells with dynamic holes. For MintAgree's 45+ pages, enable PPR for pages with mixed static/dynamic content (e.g., dashboard with user-specific data).
- **Instrumentation Hook**: Use `instrumentation.ts` for production monitoring and startup logic.
- **Turbopack**: Still experimental but faster HMR. For CI builds, use Webpack for stability.

### RSC vs Client Components
- **Rule of thumb**: If it doesn't need interactivity or browser APIs, make it a Server Component. This eliminates JS bundle for that component entirely.
- **React Three Fiber** must be in a Client Component (it needs `window`, WebGL, events). Wrap it in `'use client'` and lazy-load the scene.
- **Data fetching in RSC**: Fetch inside the component. Data is fetched on the server, HTML is streamed. No `useEffect` waterfall.
- **When to use Client Components**: Event handlers, `useState`, `useEffect`, browser APIs, third-party libraries that touch DOM directly.

### Streaming & Suspense
- Next.js 16 streams RSC output. Wrap slow components (e.g., heavy 3D scenes, analytics widgets) in `<Suspense>`.
- Use `loading.tsx` (or `loading.js`) per route segment for instant skeletons.
- **Actionable**: For React Three Fiber scenes, wrap the canvas in `<Suspense fallback={<Spinner />}>`. The fallback renders immediately, the 3D scene streams in.

### Code Example: Lazy-loading R3F Scene
```tsx
// components/SceneContainer.tsx
'use client'
import { Suspense, lazy } from 'react'

const Scene = lazy(() => import('./Scene'))

export default function SceneContainer() {
  return (
    <Suspense fallback={<div className="skeleton h-96 w-full" />}>
      <Scene />
    </Suspense>
  )
}
```

## 2. Web Vitals 2026 Targets (INP Era)

### INP Replaces FID (March 2024+)
- **Interaction to Next Paint (INP)** measures responsiveness across the entire page lifecycle, not just the first interaction.
- **Thresholds** (as of 2026, stable):
  - **Good**: INP <= 200ms
  - **Needs Improvement**: 200ms < INP <= 500ms
  - **Poor**: INP > 500ms
- FID is deprecated. Lighthouse and PageSpeed Insights now weight INP heavily.

### LCP (Largest Contentful Paint)
- **Good**: <= 2.5s
- Target for SaaS landers: Hero image or H1 must render within 2.5s.

### CLS (Cumulative Layout Shift)
- **Good**: <= 0.1
- Critical for SaaS: Avoid layout shifts in pricing tables, signup forms, and CTA buttons.

### TTFB (Time to First Byte)
- Vercel Edge caching typically yields < 100ms TTFB globally. Ensure `fetch` requests are cached or use ISR.

### Actionable for MintAgree
1. **Run Web Vitals in production** via `next/web-vitals`:
```tsx
// app/layout.tsx or app/_components/WebVitals.tsx
'use client'
import { useReportWebVitals } from 'next/web-vitals'

export function WebVitals() {
  useReportWebVitals((metric) => {
    // Send to analytics endpoint or Vercel Analytics
    console.log(metric)
  })
}
```
2. **Monitor INP** with Chrome DevTools > Performance panel. Look for long tasks blocking the main thread.
3. **Reduce INP**: Break up synchronous React renders, debounce heavy event handlers, avoid forced synchronous layouts.

## 3. React Three Fiber Optimization

### Reducing WebGL Bundle Size
- **three** is ~600KB gzipped. Use `tree-shaking` with `import { Mesh } from 'three'` instead of `import * as THREE from 'three'`.
- **drei** helpers are tree-shakeable: import only what you need (`import { OrbitControls } from '@react-three/drei'`).
- Consider **drei/MeshDistortMaterial** or lightweight shaders instead of heavy post-processing effects.

### Lazy Loading Scenes
- R3F scenes should never block initial page paint. Always lazy-load:
```tsx
const Canvas = lazy(() => import('@react-three/fiber').then(m => ({ default: m.Canvas })))
const MyScene = lazy(() => import('./MyScene'))
```
- Use `viewport` observer to render WebGL only when in view:
```tsx
import { useInView } from 'react-intersection-observer'

function LazyCanvas() {
  const { ref, inView } = useInView({ triggerOnce: true, rootMargin: '200px' })
  return (
    <div ref={ref} className="h-96 w-full">
      {inView && (
        <Suspense fallback={null}>
          <Canvas>
            <MyScene />
          </Canvas>
        </Suspense>
      )}
    </div>
  )
}
```

### Mobile GPU Limits
- Use `dpr={[1, 2]}` on `<Canvas>` to cap pixel ratio on high-DPI mobile screens.
- Limit triangle count (< 100k is safe for mid-range mobile).
- Use baked lighting/lightmaps instead of real-time shadows where possible.
- **Performance monitor**: Use `<PerformanceMonitor>` from `@react-three/drei` to downgrade effects on low-end GPUs:
```tsx
import { PerformanceMonitor } from '@react-three/drei'

<PerformanceMonitor onDecline={() => setDpr(1)}>
  <Canvas dpr={dpr}>...</Canvas>
</PerformanceMonitor>
```

### Actionable
- Add `react-intersection-observer` to MintAgree dependencies if not present.
- Audit R3F pages with Chrome DevTools > Renderer tab. Look for excessive draw calls.

## 4. Tailwind CSS v4 Performance

### Compiled CSS Strategy
- Tailwind v4 is a **CSS-first engine** powered by a new Rust-based compiler. It parses your CSS files to find `@import "tailwindcss"` and generates utility classes from that.
- No `tailwind.config.ts` needed in v4 (though backward-compatible). Configuration lives in CSS via `@theme`.
- For MintAgree: If still on v3, migration to v4 yields ~10x faster build times and smaller CSS output.

### Purging (Now "Content Detection")
- Tailwind v4 scans source files automatically. Ensure your `content` paths (or v4 equivalent) include all JSX/TSX files.
- **Unused CSS**: Tailwind v4 generates only classes found in source. No manual purge config required.

### Critical CSS Extraction
- Next.js does not yet support native critical CSS extraction for App Router in a single config toggle.
- **Workaround**: Use `@layer utilities` and `@layer components` in Tailwind. Keep above-the-fold styles in `globals.css` and defer heavy component styles.
- For v4, ensure `postcss.config.js` uses `@tailwindcss/postcss`:
```js
// postcss.config.js
module.exports = {
  plugins: {
    '@tailwindcss/postcss': {},
  },
}
```

### Actionable
- Check if MintAgree is on Tailwind v3 or v4. If v3, plan migration.
- Run `css Unused` audit in Chrome DevTools Coverage tab after login flows.

## 5. Accessibility for SaaS Landing Pages (WCAG 2.2)

### WCAG 2.2 Update (2023, now widely enforced)
- **Focus Not Obscured (2.4.11/12)**: When an element gets focus, it must not be fully/hidden by author-created content. Sticky headers and modals are frequent offenders.
- **Target Size (Minimum) (2.5.8)**: Targets must be at least 24x24 CSS pixels (except when inline or in a sentence).
- **Dragging Movements (2.5.7)**: Provide single-pointer alternatives for drag actions.
- **Consistent Help (3.2.6)**: Help mechanisms (chat, contact links) in same relative position across pages.

### Keyboard Navigation
- All interactive elements must be reachable via Tab.
- Trap focus inside modals/dialogs (`<Dialog>` from Radix or Headless UI handles this).
- Visible focus rings: `focus-visible:ring-2 focus-visible:ring-offset-2` in Tailwind.

### Focus Management
- On route change in Next.js App Router, focus should move to `<main>` or an `h1`. Use `useEffect` in a layout wrapper or a dedicated focus manager component.
- Skip links: Add `<a href="#main-content" className="sr-only focus:not-sr-only">Skip to content</a>` at the top of `layout.tsx`.

### ARIA Patterns for SaaS
- **Marketing carousels**: `role="region" aria-roledescription="carousel"` and proper `aria-live` politeness.
- **Pricing tables**: Use `<table>` with proper `<th scope="col">`. Avoid `div`-based tables.
- **FAQ accordions**: `details/summary` is native and accessible. If custom, use `button` with `aria-expanded`.
- **Toast notifications**: `role="status"` or `role="alert"` with `aria-live="polite"`.

### Actionable
- Run `axe DevTools` extension on every major page.
- Check MintAgree's pricing table markup. Ensure `<th>` elements have scope.
- Verify focus outlines are not removed by `outline-none` without a replacement.

## 6. Image Optimization for SaaS Sites

### Next.js Image
- Always use `<Image>` from `next/image` (App Router version).
- Use `priority` on hero images to preload them (boosts LCP).
- Use `placeholder="blur"` with `blurDataURL` for below-the-fold large images to prevent layout shifts.

### AVIF Format
- Next.js Image automatically serves AVIF when supported (if `formats: ['image/avif', 'image/webp']` in `next.config.ts`).
- AVIF offers ~50% smaller files than WebP at comparable quality.
- **Actionable**: Ensure `next.config.ts` includes:
```ts
const nextConfig = {
  images: {
    formats: ['image/avif', 'image/webp'],
    minimumCacheTTL: 60 * 60 * 24 * 30, // 30 days
  },
}
```

### Responsive Images
- Use `sizes` prop accurately. Incorrect `sizes` can cause Next.js to serve oversized images.
```tsx
<Image
  src="/hero.jpg"
  alt="Agreemint hero"
  fill
  priority
  sizes="(max-width: 768px) 100vw, 50vw"
/>
```

### LCP Impact
- The LCP element is often the largest image. Preload it.
- Avoid `background-image` CSS for hero sections; use `<Image fill>` instead so Next.js can optimize.
- Add `fetchpriority="high"` (handled automatically by Next.js `<Image priority>`).

### Actionable
- Inspect all pages with heavy imagery in PageSpeed Insights. Identify LCP element.
- Convert hero images from PNG/JPG to AVIF source files if using static imports.

## 7. Core Web Vitals Impact on Google Rankings

### How Much Do They Help?
- **Confirmed ranking factor since June 2021** (Page Experience update).
- For **low-competition keywords**, technical performance can provide a decisive edge because many competitors have slow sites.
- **Correlation data (2025-2026)**: Sites in the top 3 positions for < 500 MSV (monthly search volume) keywords often have "Good" CWV scores, but content relevance still dominates.
- **INP in 2026**: With INP now replacing FID, poor interactivity scores can directly hurt rankings, especially for informational queries where users bounce quickly.

### SaaS-Specific Context
- For MintAgree (contract/SaaS niche), low-competition long-tail keywords (e.g., "simple contract management for freelancers") are winnable with good content + fast performance.
- However: if a page has poor CLS and a user clicks "Sign Up" but the button shifts, the bounce rate increases, indirectly lowering dwell time signals.
- **Actionable**: Prioritize LCP and CLS over FCP. A fast LCP on pricing pages directly correlates with conversion, and conversion rate is an implicit user-satisfaction signal.

### Summary for Mercury
- CWV won't outrank high-DR sites with better content, but for niche/low-competition terms, being the only "Good" performer in the SERP is a tangible advantage. Treat CWV as a tiebreaker and conversion booster, not a silver bullet.

## 8. CSS-in-JS vs Compiled CSS in 2026

### The Consensus
- **CSS-in-JS is losing favor** in Next.js App Router contexts.
- **Why**: Most CSS-in-JS libraries (styled-components, emotion) require a runtime. In the React Server Components (RSC) world, injecting styles at runtime is problematic because RSCs don't execute in the browser.
- **Zero-runtime CSS-in-JS** (like Vanilla Extract, Linaria, Pigment CSS) is viable but adds build complexity.

### Current Best Practice
- **Tailwind CSS** (compiled, atomic, zero-runtime) is the dominant choice for new Next.js projects in 2026.
- **CSS Modules** remain a solid, zero-runtime alternative for component-scoped styles.
- **Recommendation for MintAgree**: Stick with Tailwind CSS. If custom complex animations require CSS-in-JS, use CSS Modules or Tailwind's `@layer` features. Avoid `styled-components` in new App Router pages.

### Performance Implications
- Runtime CSS-in-JS adds ~10-20KB to bundle and forces style recalculation on hydration.
- Tailwind v4 outputs a single static CSS file, fully cacheable, with no runtime overhead.

### Actionable
- Audit MintAgree for any `styled-components` or `@emotion` usage. Migrate to Tailwind or CSS Modules where possible.
- Use `postcss` to inline critical CSS if needed.

## 9. Bundle Analysis for Next.js Apps

### Finding Dead Code
1. **@next/bundle-analyzer**:
```bash
npm install -D @next/bundle-analyzer
# next.config.ts
const withBundleAnalyzer = require('@next/bundle-analyzer')({
  enabled: process.env.ANALYZE === 'true',
})
module.exports = withBundleAnalyzer(nextConfig)
```
Run: `ANALYZE=true npm run build`

2. **Webpack Bundle Analyzer** generates interactive treemaps. Look for:
   - Large `node_modules` (e.g., `lodash` -> replace with `lodash-es` or per-function imports).
   - Multiple versions of the same package.
   - `three.js` taking > 30% of client bundle.

3. **next-bundle-analyzer alternative**: Use `webpack-bundle-analyzer` directly if not using Next.js' wrapper.

4. **Coverage in DevTools**: Chrome DevTools > Coverage tab. Load a page, stop recording, see unused bytes highlighted in red.

### Eliminating Dead Code
- Use `import { debounce } from 'lodash-es'` instead of `import _ from 'lodash'`.
- For R3F, ensure `three` is not accidentally imported into RSC bundles. Mark R3F components as `'use client'` and trace imports.
- Tree-shake `@react-three/drei`: `import { Html } from '@react-three/drei'` not `import * as Drei from '@react-three/drei'`.

### Actionable
- Run bundle analyzer before every major release.
- Set a bundle budget: target < 200KB initial JS for landing pages. R3F pages may be higher, but lazy-load the canvas.

## 10. Edge Caching Strategies for Vercel

### What Gets Cached
- **Static files** in `.next/static/**`: Cached indefinitely (hashed filenames).
- **ISR pages** (`revalidate`): Cached at Edge. First request after stale-while-revalidate triggers background regeneration.
- **Images** (`/_next/image`): Cached according to `minimumCacheTTL` in `next.config.ts`.
- **Serverless Function responses**: Only if you explicitly set `Cache-Control` headers.

### What Doesn't Get Cached
- **API Routes** (`app/api/**`) without explicit `Cache-Control`.
- **Dynamic routes** without `generateStaticParams` or ISR.
- **Non-GET requests** (POST, PUT, etc.).
- **RSC payloads** (`/_next/data`): Cached differently; generally short TTL unless ISR is configured.

### Structuring for Maximum Cache Hits
1. **Prefer Static Generation + ISR** for marketing pages. If data changes daily, use `revalidate: 86400`.
2. **Use `unstable_cache` for DB queries** in Server Components (Next.js 14+). This caches the DB call result across requests and deployments (duration-based):
```tsx
import { unstable_cache } from 'next/cache'

const getPosts = unstable_cache(
  async () => db.query.posts.findMany(),
  ['posts'],
  { revalidate: 3600, tags: ['posts'] }
)
```
3. **Fetch cache**: `fetch` in RSC is automatically memoized per-request. Add `next: { revalidate: 60 }` for cross-request caching.
4. **Headers for API routes**:
```ts
// app/api/pricing/route.ts
export async function GET() {
  const data = await getPricing()
  return Response.json(data, {
    headers: {
      'Cache-Control': 'public, s-maxage=60, stale-while-revalidate=300',
    },
  })
}
```

### Vercel Edge Network nuances
- **Edge Functions** (`export const runtime = 'edge'`) run closer to the user but have restrictions (Node.js APIs, bundle size <= 1MB).
- **Middleware** runs at Edge. Keep it lightweight; heavy middleware adds TTFB.

### Actionable
- Review `app/**/page.tsx` files. Any static content without `revalidate` or `generateStaticParams` is generating on-demand serverless runs. Convert to static or ISR.
- Set a `minimumCacheTTL` of at least 1 day for images.
- Use `unstable_cache` for Drizzle ORM queries that are frequently accessed but rarely change (e.g., feature list, pricing tiers).
